Chat vs Agent：差别到底在哪

用同一个问题做两遍，亲手感受普通 LLM 与 Agent 的本质差别。

1你将学会

勾选完成情况会自动同步到首页学习进度。

  • 我能用一个具体例子说清"Chat 和 Agent 差在哪"，不用任何术语
  • 我能识别一个工具是不是真正的 Agent（看它有没有"看-想-做-再看"的循环）
  • 我能判断"这个任务该用 Chat 还是该用 Agent"

2背景小知识

把 LLM 想象成一个只会动嘴、不会动手的超级实习生。

  • 普通 Chat：你问一句，它答一句。它聪明但被关在屋子里——不能上网、不能写文件、不能点按钮。
  • Agent：同一个实习生，但给了它键盘、浏览器、文件柜的钥匙。你给它一个目标，它会自己决定先做什么、再看结果、再决定下一步，直到完成或卡住。

差别不在模型大小，而在"它能不能动手 + 能不能多步连贯地动"。

核心公式：Agent = LLM × 工具 × 自主循环。后面所有讲义都建立在这一个直觉上。
📖 5 个核心词建立 Agent 直觉（点开速读，不必现在搞懂全部）

1. Agent（智能体） · 能自己"看-想-做-再看"的 LLM。给目标→主动调工具→看结果→决定下一步，直到完成或卡住。把它当成需要你把目标讲清楚的实习生。

2. Tool（工具） · Agent 的"手脚"。读写文件、搜网、跑命令、调企业 API。模型本身只会"想"，能不能"做事"全靠 Tool。

3. Skill（技能 / 工作流封装） · 把"提示词 + 步骤 + 约束 + 产物路径"打包，让 Agent 下次稳定复现。Skill 之于 Agent，就像 SOP 之于团队——是 Agent 时代真正可累积的生产资料。

4. Context（上下文） · 这一次回答时它"能看到的所有信息"。窗口有限会被挤掉。当你觉得 Agent "变笨了"，多半不是模型差，而是上下文管理不好。L4 会教解法。

5. RAG / LLM Wiki · 让 Agent 每次回答前先去外部知识库查一下再答。与其每次粘贴背景，不如一次性写进 llm-wiki/。L4 会带你建一个最简版。

进阶概念（先不用懂）：MCP（L6/L8）· Plan/Spec Mode（L9）· Human-in-the-loop（L10）· Eval（L11+）· Multi-Agent（进阶选读）。

① 选定一款 Agent 客户端

任选其一，公司允许即可。本手册的 prompt 在三者上都验证过：

客户端适合谁备注

Claude Code喜欢命令行、要操作较多本地文件Skill 体系最成熟

② 工作目录

在常用工作盘下，建一个空目录作为整门课的根：

pm-agent-lab/

每一讲都会在它下面新增子目录，不要和别的项目混在一起。强烈建议放在你日常会备份的位置（iCloud / OneDrive / 公司云盘任选）。

③ 企业办公 CLI（C 篇章必选，A/B 不强制）

口径说明：本手册以飞书 lark-cli 为示例工具，因为它在字节内部最完整、覆盖 doc / sheets / im / calendar / base / wiki。如果你的公司用钉钉、企微、Notion、Slack、Microsoft 365 等其他办公套件，把对应的 CLI / Open API / 官方 SDK 同等代入即可——课程讲的是"如何让 Agent 调企业能力"这件事，不绑定某一家厂商。

C 篇章（L6–L9）会大量使用企业原生能力（文档 / 表格 / 即时消息 / 日历 / 多维表 / 知识库）。以 lark-cli 为例的最小安装步骤：

  1. 按公司内文档安装 lark-cli（其他套件请安装对应 CLI / SDK）
  2. 第一次使用前执行：lark-cli config init
  3. 登录授权：lark-cli auth login
  4. 验证：随便跑一个只读命令，例如查一下今天的日程

遇到 Permission denied 几乎都是 scope/权限范围不全；按提示加 scope、重新授权即可。其他套件的等价问题通常也是"OAuth 授权范围不够"，处理思路一致。

④ 常见踩坑

现象多半原因处理
Agent 一直说"我无法访问外网"客户端没开联网 / 没装 web 工具切到带联网能力的模式
Agent 说"我没有权限写入文件"客户端在受限模式启用"允许读写本地文件"权限
同样的 prompt 结果差异很大上下文不一致 / 模型随机性把 prompt 写得更精确（L3 会教）
lark-cli 报 Permission deniedscope 不全重新跑 auth login 并加上提示的 scope

4实战任务

我们让两个窗口完成同一件事。注意观察的不是答案本身，而是它们处理问题的方式。

1在 Chat 窗口（窗口 A）问一遍

🎯你要做什么

让普通 Chat 给你"过去一周的素材整理"——根据你的练手场景。

⌨️你输入什么
帮我把下面这一周散落在各处的需求线索（飞书 IM 提到的、邮件里收到的、用户访谈摘录），
整理成 5–10 条候选需求，每条带：需求一句话描述、来源、初步价值判断。
按重要性排序。

[这里你想象贴入：3-5 段杂乱的来源原文]
帮我从下面这一周的 3 段会议纪要里，提取出 5–10 个待办与决策事项，
每条带：事项一句话描述、归属人（如纪要里有提到）、紧急度。
按紧急度排序。

[这里你想象贴入：3 段会议纪要全文]
帮我整理过去一周 AI Agent 行业的 5–10 条重要新闻，每条带：标题、来源、一句话摘要。
按重要性排序。
👀预期看到什么
  • 它会直接给一段 markdown 列表，看起来很整齐。
  • 仔细看会发现一个或多个问题：时间不准、来源含糊、链接是编的、问"刚才那些来源能确认吗"它会道歉但还是给不出真链接。
⚠️ 偏差纠偏
· 它说"我无法访问实时信息"——这就是答案，恰恰证明它是普通 Chat。继续下一步。
· 它居然给了带链接的真实新闻 → 多半是它接了搜索插件，已经偏 Agent 了。换一个不带搜索的更纯粹的 Chat 再试。

2在 Agent 窗口（窗口 B）问一遍

🎯你要做什么

同样的问题给 Agent，但先什么都不要补充，看它自己怎么动。

⌨️你输入什么
请你扮演 PM 助理。读取我桌面上的这一周需求来源原文（我把它们汇总在
pm-agent-lab/backlog-lab/L0-demo/sources.md 里，你可以直接打开读），
帮我整理成 5–10 条候选需求，每条字段：
- 需求一句话描述（动词+宾语+受益对象）
- 来源（具体到哪条飞书消息 / 哪段访谈）
- 初步价值判断（1-2 句话）
- 我建议的下一步动作
按"价值大、待办近"两条线综合排序。
请你扮演 PM 助理。读取我桌面上 pm-agent-lab/meeting-lab/L0-demo/notes/
目录下这一周所有 .md 会议纪要，提取出 5–10 个待办与决策事项，每条字段：
- 事项一句话描述
- 归属人（纪要里若没有，就标"待指派"）
- 紧急度（高/中/低）+ 截止建议日期
- 该事项关联的会议名/日期
按紧急度排序，并在末尾告诉我"哪几条没有归属人需要我决定"。
帮我调研过去 7 天 AI Agent 行业的 5–10 条重要新闻，每条带：标题、来源链接、发布日期、一句话摘要。
按重要性排序。
👀预期看到什么
  • 它不会立刻给答案，而是会先做一连串动作：调用 web search / read file / list dir，"我搜到 X 条，正在整理"。
  • 最终给你的列表里，链接可点、日期真实、来源具体。
📝 重点观察：Agent 在动手之前你能看到它的"思考-动作-观察"循环。这就是它和 Chat 的根本差别。

3让 Agent 把结果"存到本地"——这是 Chat 完全做不到的

🎯你要做什么

让 Agent 把刚才的清单写到你电脑的指定目录。这是本讲最关键的一步——让你亲眼看到 Agent 在动手。

⌨️你输入什么
把刚才整理的需求候选清单写到我本地：
- 路径：pm-agent-lab/backlog-lab/L0-demo/backlog.md
- 格式：markdown 表格，列分别是 需求描述 / 来源 / 价值判断 / 下一步 / 提出日期
- 如果目录不存在，请先创建
完成后告诉我你创建/修改了哪些文件。
把刚才整理的待办与决策清单写到我本地：
- 路径：pm-agent-lab/meeting-lab/L0-demo/follow-up.md
- 格式：markdown 表格，列分别是 事项 / 归属人 / 紧急度 / 截止日期 / 来源会议
- 如果目录不存在，请先创建
完成后告诉我你创建/修改了哪些文件。
把刚才整理的新闻清单写到我本地：
- 路径：pm-agent-lab/news-lab/L0-demo/news.md
- 格式：markdown 表格，列分别是 标题 / 来源 / 日期 / 摘要 / 链接
- 如果目录不存在，请先创建
完成后告诉我你创建/修改了哪些文件。

🤯 这一步要让你"破防"

现在回到窗口 A，让那个普通 Chat 也"把结果写到我本地 pm-agent-lab/..."。
它会很有礼貌地告诉你它做不到——它只能在对话框里输出文字，没有手。
这就是 Chat vs Agent 最直观的差别。

4让 Agent 自己"复盘"刚才做了什么

让它用自己的话总结一遍它刚才的动作链。这个总结你后面会用到，三个场景通用。

⌨️你输入什么
请用 5 行以内回答：
1) 你刚才一共调用了哪些工具？
2) 它们是按什么顺序调用的？
3) 在哪一步你做了一次"看到结果再决定下一步"的判断？
4) 这件事如果不让你联网/不让你写文件，能完成吗？
5) 这是 Chat 做不到、但你能做到的核心原因是什么？

把这段回答保存到 <场景>-lab/L0-demo/agent-vs-chat.md，作为本讲学习笔记。

5自检清单

  • 我能拿出 backlog.md 这个文件给同事看
  • 我能用 30 秒讲清"Chat 和 Agent 差在哪"，例子就是我刚才做的这件事
  • 我能判断下面三个任务该用 Chat 还是 Agent（解释法规条文 / 把法规要点存表格到本地 / 跨竞品调研写到 PRD）

🧠 快速自测

Q1. 下列哪个最准确地描述 Agent 与 Chat 的差别？
正确答案 C：Agent 的本质是 LLM × 工具 × 自主循环，模型大小和是否联网都不是充要条件。
Q2. 下面哪一个任务必须用 Agent？
正确答案 B：B 同时需要"联网调研 + 写本地文件"两类工具，Chat 都做不到。其余三项纯文本任务，Chat 即可。

6思考题 & 下一讲引子

思考题

  • 回看 Step 2 的 prompt，如果你是 Agent，你会怎么决定搜哪些关键词？这反映：你给目标越清晰，Agent 走弯路越少。
  • 在你过去一周的工作里，有哪 3 个任务是你曾经"问过 Chat 但发现只能给空话"的？记下来，L12 时回来检视。
