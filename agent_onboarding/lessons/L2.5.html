让 Agent 操作浏览器：突破 API 与公开数据的边界

当数据在登录态后面、当任务是"重复点点点"、当系统没有 API——让 Agent 接管你的浏览器。

1你将学会

勾选完成情况会自动同步到首页学习进度。

  • 我能让 Agent 真的打开一个网页、登录、点击、填写、截图、抓数据
  • 我能判断哪些任务该走"浏览器操作"、哪些该走"API/CLI"
  • 我能为浏览器操作任务写一份安全清单：不让 Agent 误点破坏性按钮

2背景小知识

到 L2 为止，你让 Agent 做的所有事都属于"模型 + 文件系统 + 公开网页搜索"。但 PM 工作里很多场景不在公开互联网上：

  • 公司后台运营数据：必须登录才能看
  • 飞书文档列表 / 多维表格筛选结果：你自己看着比 API 更直观
  • 竞品 SaaS 的功能演示：公开页就能点，但要真点一遍才知道交互长什么样
  • 各种"重复点 30 次"的体力活：批量下载附件、批量改字段、批量发送邀请

浏览器操作（Browser Use）就是给 Agent 一个真实浏览器的钥匙。它能像人一样：打开网页 → 看到页面截图 → 决定点哪 → 真的点下去 → 看新页面 → 继续……

💡 直觉口诀

没 API 的地方，浏览器就是兜底 API。
⚠️ 重要前置：浏览器操作 = Agent 真的在你电脑上点击。所以风险也最大——它点错按钮和你点错没区别。本讲会专门给你一份"安全清单"。

🧭 互动决策器：Browser Use 还是 API？

回答 3 个问题，告诉你这个任务该走哪条路。可以反复重置——这就是 PM 拿到一个新任务时心里要过的清单。

1有没有 API 2要不要登录 3会不会写 ✓建议
decision://browser-vs-api

📝 这一步教会你什么

人在环路：登录 / 验证码 / 二次验证一律由人来做，Agent 只做之后的事。
断点协作：Agent 主动停下等你确认，是浏览器操作的标配姿势。
⚠️如果出错怎么办
  • 偏差 A：Agent 自作聪明开始填密码 → 立刻打断：
    停下，不要替我登录。把当前页留在登录页，等我说"我登录好了"再继续。后续所有需要凭证的步骤都同样处理。
  • 偏差 B：登录后跳到了"个人版"而不是"企业版" → 让它读 URL 自检：
    请先 print 出当前页面的 URL 和 title 给我看，确认是不是企业空间。如果不是，停下来问我怎么切。

3让 Agent 抓取列表数据（核心）

💡 节奏：这一步是本讲价值最高的一步——让 Agent 把"肉眼能看到的列表"变成结构化数据。完成它你就解锁了"登录态后的世界"。
⌨️你输入什么
现在抓取当前 IM 搜索结果页里所有可见的消息条目，输出成结构化数据：
- 字段：snippet（消息片段）/ sender（发件人）/ chat_name（来自哪个群或私聊）/ sent_at / url（消息深链）
- 滚动到底加载更多，但只抓最近 14 天内的；早于 14 天就停
- 写入：pm-agent-lab/backlog-lab/L2.5-demo/data/raw-leads.json
- 同时给我一段文字汇报：本次共抓 N 条 / 各 chat_name 分布 / 滚动了几次

约束：
1) 不要点开任何消息进入聊天详情，只在搜索结果列表页读取
2) 不要做任何会改数据的操作（不要回复、点赞、转发）
3) 抓完发我看，不要自动进入下一步
现在抓取当前会议/妙记列表页里所有可见的条目，输出成结构化数据：
- 字段：meeting_title / type（视频会议/纪要/线下会）/ host（主持人）/ start_at / duration_min / url
- 滚动到底加载更多，但只抓最近 7 天内的；早于 7 天就停
- 写入：pm-agent-lab/meeting-lab/L2.5-demo/data/recent-meetings.json
- 同时给我一段文字汇报：本次共抓 N 条 / 各 type 分布 / 滚动了几次

约束：
1) 不要点开任何会议/纪要进入详情页，只在列表页读取
2) 不要做任何会改数据的操作（不要重命名、删除、修改参会人）
3) 抓完发我看，不要自动进入下一步
现在抓取"最近"页里所有可见的文档条目，输出成结构化数据：
- 字段：title / type（文档/表格/多维表格/Wiki/其它）/ owner / last_visited_at / url
- 滚动到底加载更多，但只抓最近 30 天内打开的；早于 30 天就停
- 写入：pm-agent-lab/news-lab/L2.5-demo/data/recent-docs.json
- 同时给我一段文字汇报：本次共抓 N 条 / 各 type 分布 / 滚动了几次

约束：
1) 不要点开任何文档，只在列表页读取
2) 不要做任何会改数据的操作（不要点收藏/取消收藏/删除）
3) 抓完发我看，不要自动进入下一步
👀预期看到什么
  • 真的产生结构化 JSON 文件，字段齐全
  • 一段汇报：滚动了 X 次、抓了 N 条、最早一条是什么时间
  • Agent 停在原地等你确认
⚠️如果出错怎么办
  • 偏差 A：滚动一次就停了 → 纠偏：
    你只滚了 1 次，但页面下方还有更多内容。请继续滚动直到出现"没有更多"或最早条目超出时间范围为止；每次滚动后等 1 秒让数据加载完。
  • 偏差 B：URL 字段抓的是相对路径 → 纠偏：
    url 必须是完整的可点击链接。如果只能拿到相对路径，请基于 location.origin 拼成完整 URL。
  • 偏差 C：抓到了重复条目 → 纠偏：
    按 url 去重，重复的只保留时间最新那条。给我去重前后各多少条。

🎯你要做什么

把刚抓到的数据用上——这是 PM 视角才会想到的二次加工。

⌨️你输入什么

4) 一句话洞察：从这批线索看，过去两周用户/同事最关心的是什么主题？

5自检清单

  • 我能在浏览器里看到 Agent 真的在动（鼠标移动 / 点击高亮）
  • 我能让它在登录页停下来等我手动操作
  • 我能从结构化 JSON 文件里看出它真抓到了登录态数据
  • 我有一份 BROWSER_SAFETY.md，并能 30 秒讲清"哪些事它不能独立做"
  • 我能判断一个新任务该走浏览器操作还是 API/CLI（提示：有 API 永远优先 API）

🧠 快速自测（呼应"三问"）

Q1. 同一个任务既能走 API 也能走 Browser Use，应该优先？
正确答案 B：有 API 永远优先 API。Browser Use 是"没 API 时的兜底"，不是首选。
Q2. 浏览器任务要登录，PM 该怎么做？
正确答案 C：人在环路是浏览器操作的铁律。凭证 / 二次验证一律自己来，Agent 只做之后的事。
Q3. 只读浏览 vs 写操作（点删除/发送/支付），应该怎么处理？
正确答案 B：写操作（POST/PUT/DELETE）默认要人工确认，是浏览器操作的安全护栏；只读任务再放手让 Agent 自己跑。

8副线练习（可选）

本讲主线已可在三个场景切换，副线建议挑战更进阶的玩法。把"浏览器抓登录态数据 → 落本地 → 出报告"这套姿势用在你真实的工作场景上：

  • 竞品 SaaS 公开页全功能截图：访问竞品官网 + 公开 demo 站，让 Agent 把"产品功能列表 / 各 Tab 截图 / 关键交互 GIF"批量抓下来归档
  • 公司运营后台日常巡检：让 Agent 每天抓你最关心的几个核心指标看板截图，存成"日巡检包"
  • 批量整理网页书签 / 飞书收藏夹：让 Agent 把你混乱的收藏全部分类、去重、写成一份索引
💡 副线产物挂在对应 lab 目录，命名保持和主线一致——这样 L5 一次封装就能服务所有副线场景。

9给 PM 的小结：浏览器操作的"三问"

任何浏览器任务开始前，先问自己：

问怎么决策为什么
1. 有没有 API？有就走 API更稳、更便宜、更安全、可重跑
2. 要不要登录？人工登录 + Agent 接管凭证不喂模型；二次验证人工来
3. 会不会改东西？写操作默认人工确认只读任务再放手让 Agent 自己跑

🎯 一句话带走

三问做对，浏览器操作就从"高风险炫技"变成"PM 日常工具"。
