L4.5 · 模块 B 提效进阶 ⏱ 50 min 难度 ⭐⭐⭐ 当前练手场景：行业 / 竞品周报

多模态：截图 / 设计稿 / 报错图

PM 每天处理的图比文字多 10 倍——把 Agent "看图说话"的能力打开，是 ROI 最高的能力升级之一。

1你将学会

勾选完成情况会自动同步到首页学习进度。

  • 我能把竞品截图丢给 Agent，让它逐元素拆解 UI 与文案
  • 我能让 Agent 对比两版设计稿并自动写"改稿说明"
  • 我能把研发或读者发来的报错截图直接交给 Agent 解读

2背景小知识

到 L4 你已经会"喂文字"。但你看：

  • 竞品同事甩来一张图：".png 放群里了，看一眼？"
  • 设计师改稿：3 张稿子飞过来，让你说"哪版好"
  • 研发 / 客服报问题：截一张报错图

这些场景，文字描述能力都是次优解——直接喂图最快。多模态就是：Agent 不仅能读文字，还能读图，甚至读 PDF / 视频帧 / 音频。

💡 直觉口诀

能截图就别打字。你的眼睛已经看到了——让 Agent 也"看到"，比你转译一遍快 10 倍。
⚠️ 红线：图里如果有用户隐私 / 公司机密数字 / 未公开人脸——先脱敏再喂模型，特别是外部模型。

🛠 互动模块：图的四类用法决策器

把这张图想象成你刚刚收到的——选一个最像你场景的，看推荐 prompt 模板。顶栏切换三场景时，每张卡片下的"典型场景示例"会同步换文案。

🔍
拆解
把一张竞品 / UI 图变成结构化文字
"逐模块描述" / "列出 CTA 文案与视觉权重"
典型场景：拆解竞品页面截图
🆚
对比
多张图找差异，自动写"改稿说明"
"变了什么 / 推断意图 / 应追问的问题"
典型场景：旧版 vs 新版设计稿对比
🌐
解读
把一张图翻译给不同受众
"翻译成非技术语言 / 给同步建议"
典型场景：把报错截图讲给非技术同事
🔁
反向构造
从图反推产物（PRD / JSON / 大纲）
"把白板转成 PRD 目录 / 把表格图转 JSON"
典型场景：白板照片直接转 PRD 草稿
推荐 prompt 骨架

        
⚠ 何时不要用这个：

      

      

        🚧 红线提示：图里如果有用户隐私 / 公司机密数字 / 未公开人脸——先脱敏再喂模型。把人脸打码、把订单号 / 用户 ID / 金额抹掉，再让 Agent 看；外部模型尤其敏感。

1让 Agent 拆解一张竞品截图

        
🎯你要做什么

        
把一张竞品页面截图扔进去，让 Agent 像产品分析师一样拆解。

        
⌨️你输入什么（同时附上截图）

        

          

            
            
            
            
            
          

          
[附图：竞品某页面截图]

[角色] 你是资深 PM，正在做竞品分析。

[上下文] 这是 <竞品名> 的 <页面名> 截图。我想看清楚他们在这个页面想做什么。

[约束]
- 只描述图里真实存在的元素，不要脑补图外的功能

- 不要写"建议我们也做"这种空话；我自己来判断

[产物]
请输出到 pm-agent-lab/vision-lab/competitor-<竞品名>-<页面名>.md，分 4 段：
1) 页面信息架构（顶到底各模块）
2) 每个 CTA 按钮的文案 + 视觉权重排序
3) 核心数据/卖点的呈现方式（颜色、留白、字号优先级）
4) 你能从布局推断出的"核心目标动作"是什么（不超过 1 句话）

          
[附图：飞书 IM / 邮件里同事发来的需求截图]

[角色] 你是资深 PM，正在做需求池入口的清洗。

[上下文] 这是同事在 IM 里甩给我的一段需求描述截图。我想把它结构化成需求池的一条候选项。

[约束]
- 只提取图里真实出现的文字，不要脑补需求方没写的诉求
- 看不清的字直接说"看不清"
- 不要写"建议我们立刻做"这种结论；优先级我自己来判断

[产物]
请输出到 pm-agent-lab/backlog-lab/L4.5-demo/backlog-from-im.md，分 4 段：
1) 提需人 + 提出时间（图里能看到的就抄）
2) 用户故事一句话（"作为 X，希望 Y，以便 Z"）
3) 隐含的成功指标 / 价值主张（仅基于图里文字推断）
4) 我作为 PM 应该追问需求方的 3 个问题

          
[附图：会议白板 / 投屏 / 会议纪要 OCR 截图]

[角色] 你是资深 PM，正在做会议结论的结构化沉淀。

[上下文] 这是我刚开完会拍下来的白板（或者投屏截图）。我想把它转成可跟进的待办清单。

[约束]
- 只识别图里真实出现的文字 / 箭头 / 框选关系，不要脑补未画上去的内容
- 看不清的字 / 涂改部分直接说"看不清"

[产物]
请输出到 pm-agent-lab/meeting-lab/L4.5-demo/whiteboard-<日期>.md，分 4 段：
1) 白板的视觉分区（左/中/右，或者上/下）
2) 每个分区的文字内容（OCR 转写）
3) 推断出的"决策项 / 待办项 / 风险项"分类列表

👀预期看到什么

        
    
          
  • 一份非常具体的拆解，比你"用眼睛过一遍"细很多
    • 
          
  • Agent 会准确指出"主 CTA 是橙色按钮、文案是 X、占据 Y% 视觉权重"
    • 
        


        
⚠️如果出错怎么办

        
    
          
  • 偏差 A：Agent 开始猜未在图里的功能 → 纠偏：
    你刚才提到"应该还有筛选功能"——图里没有。请只描述图里真实存在的元素，没看到的写"未在截图中"。
    • 
          
  • 偏差 B：图太糊 / 字看不清 → 让它老实承认：
    哪些元素你看不清？列出来，让我决定要不要重新截图。

2让 Agent 写"改稿对比说明"

        
这是 PM 高频场景：设计师给两版稿，你要发同步说"改了什么、为什么、是否同意"。

        
🎯你要做什么

        
喂两张设计稿（旧版 + 新版），让 Agent 自动出对比说明。

        
⌨️你输入什么（附两张图）

        

          

            
            
            
            
            
          

          
[附图 1：旧版设计稿 v1]
[附图 2：新版设计稿 v2]

[角色] 你是产品经理，正在为设计改版写一份对比说明，发给跨职能同事看。

[上下文] 这两张是同一页面的旧版（v1）和新版（v2）。我没告诉你设计师为什么改，请你自己从图里推断。

[约束]
- 客观描述变更，不要过早评价好坏
- 每个变更点写 3 行：变了什么 / 推断意图 / 我作为 PM 应该追问设计的 1 个问题
- 改动不到 5 处也老实说，不要凑数

[产物]
写到 pm-agent-lab/vision-lab/design-diff.md，结构：
- 总览（一句话：本次改版核心方向是什么）
- 变更点 1..N（每点 3 行）
- 待与设计确认的 3 个最关键问题

          
[附图 1：上周飞书 IM 提的需求截图]
[附图 2：本周同一个人/同一群里二次澄清的需求截图]

[角色] 你是产品经理，正在为同一条需求的"前后两次描述"做对照，避免漏接最终诉求。

[上下文] 这两张截图是同一个需求方在不同时间点对同一个需求的描述。我没告诉你他们为什么改了说法，请从图里推断。

[约束]
- 客观描述差异，不要急着替需求方做决定
- 每个差异点写 3 行：变了什么 / 推断意图 / 我作为 PM 应该追问需求方的 1 个问题
- 实质差异不到 3 处也老实说，不要凑数

[产物]
写到 pm-agent-lab/backlog-lab/L4.5-demo/requirement-diff.md，结构：
- 总览（一句话：本次需求口径核心变化是什么）
- 差异点 1..N（每点 3 行）
- 待与需求方确认的 3 个最关键问题

          
[附图 1：上次会议白板 / 投屏截图]
[附图 2：本次会议白板 / 投屏截图]

[角色] 你是产品经理，正在为同一议题的两次会议做"决策演进对照"，避免大家忘记上次共识。

[上下文] 这两张截图是同一议题前后两次会议的白板/投屏。我没告诉你这中间发生了什么，请从图里推断。

[约束]
- 客观描述差异，不要替参会人下结论
- 每个差异点写 3 行：变了什么 / 推断意图 / 我作为会议主持应该在下次会上澄清的 1 个问题
- 实质差异不到 3 处也老实说，不要凑数

[产物]
写到 pm-agent-lab/meeting-lab/L4.5-demo/decision-diff.md，结构：
- 总览（一句话：议题核心走向变化）
- 差异点 1..N（每点 3 行）
- 待在下次会议上澄清的 3 个最关键问题

        

        
👀预期看到什么

        
    
          
  • 一份直接可发评审群的对比说明
    • 
          
  • 每个变更点都附"我应该追问设计的问题"——这是 PM 真正的价值

2) 我说"读第 X 章"，你才读取并总结
3) 总结时区分"原文事实"和"你的推断"
不要一上来就把整份 PDF 都读完。

1) 提取文字
2) 打标签（性能 / 体验 / 价格 / 其它）
全部贴完我说"开始聚类"，你才合并分析。这样不会信息过载。

          
我会陆续贴 N 张需求来源截图（飞书 IM / 邮件 / 客服转接）。每张你只做：
1) 提取文字 + 提需人 + 提出时间
2) 打标签（功能新增 / 体验优化 / bug 修复 / 数据需求 / 其它）
全部贴完我说"开始聚类"，你才合并成 backlog 草稿。这样不会信息过载。

          
我会陆续贴 N 张会议截图（白板 / 投屏 / 聊天关键帧）。每张你只做：

2) 打标签（决策 / 待办 / 风险 / 共识 / 待澄清）
全部贴完我说"开始聚类"，你才合并成跟进清单。这样不会信息过载。

🧠 快速自测

        

          
Q1. 竞品 UI 截图扔给 Agent，最该用哪一类用法？

          

            
            
            
            
          

          
正确答案 A：单张 UI 图想"看清楚他们在干什么"，本质是把图变成结构化文字，这就是拆解。对比要至少两张图、解读偏翻译给外行、反向构造是从图反推产物。

        

        

          
Q2. 旧版 vs 新版设计稿，让 Agent 自动出"改了什么"说明？

          

            
            
            
            
          

          
正确答案 B：多张图找差异 = 对比。关键 prompt 词："变了什么 / 推断意图 / 应追问的问题"。

        

        

          
Q3. 图里出现疑似用户隐私（手机号、订单号、未公开人脸）时应该？

          

            
            
            
            
          

          
正确答案 C：红线就是"先脱敏再喂模型"——尤其外部模型。打码、抹掉数字、贴黑条都行，不要把信任建立在"提示词约束"上。

9给 PM 的小结：图的"四类用法"

      

        
用法典型动作关键 prompt 词

        
          
🔍 拆解把图变成结构化文字"逐模块描述" / "列出 CTA 文案与视觉权重"

          
🆚 对比多张图找差异"变了什么 / 推断意图 / 应追问的问题"

          
🌐 解读把图翻译给不同受众"翻译成非技术语言 / 给同步建议"

          
🔁 反向构造从图反推产物"把白板转成 PRD 目录 / 把表格图转 JSON"

        记住这四类，90% 的 PM 看图场景都能套上 Agent。看到图，先问自己："我要拆 / 比 / 译 / 反推？"
