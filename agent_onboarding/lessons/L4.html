上下文管理：搭建你的 LLM Wiki

让 Agent 永远"记得"你是谁、产品是什么、偏好怎样——把"每次都要重讲背景"这个最大隐性成本一次性消掉。

1你将学会

勾选完成情况会自动同步到首页学习进度。

  • 我能搭一个最小可用 llm-wiki/，让 Agent 每次回答前先读它
  • 我能区分什么进 Wiki、什么进 Skill、什么留在对话——三者各司其职
  • 我能用 1 句话让 Agent 把"今天会议得到的新结论"自动写回 wiki

2背景小知识

到 L3 你写出了好 prompt，但你会发现"上下文"那一段每次都在重复——"我是 PM、产品叫 X、leader 偏好简洁……"。

LLM Wiki = Agent 的私人维基：把这些长期不变 / 缓慢变化的事实写进一份结构化目录，让 Agent 每次开工前先去查一遍。

它解决三件事：

  1. 少打字：你不再每次重写背景。
  2. 少幻觉：Agent 看了真实事实，比凭"记忆"靠谱。
  3. 可演进：新结论 / 新偏好可以写回 wiki，越用越准。

💡 直觉口诀

对话是临时的，Skill 是流程，Wiki 是事实。
Wiki 给 Agent "我是谁"，Skill 给 Agent "我怎么做"，对话给 Agent "我现在要什么"。

🧠 互动模块：Wiki / Skill / 对话 三件套分类器

点击每条信息，判断它应该进哪一类。判错会变红、判对会变绿，并展开一句解释——把这一题做完，你就会用直觉感受到三者边界。

4实战任务

本讲会把一个空目录从 0 长到"Agent 真的会先读它再回答"。我们要建立的最小目录：

📁 最小 wiki 目录

pm-agent-lab/llm-wiki/
├── README.md         # 索引：这本 wiki 有什么、怎么用、何时更新
├── me.md             # 我是谁（岗位、leader、汇报偏好）
├── product.md        # 我的产品：核心定位、关键模块、当前阶段
├── glossary.md       # 我们公司/团队的"黑话词典"
├── stakeholders.md   # 主要协作方（研发、设计、运营、外部合作方）
└── memory.md         # 自由记忆：随时被 Agent 写入的临时事实

1让 Agent 帮你"采访"自己

💡 不要自己闭门写 wiki——会写得太"官方"。让 Agent 当记者来采访你，速度快 5 倍，且更接近你真实工作语境。
🎯你要做什么

让 Agent 用一问一答的方式，把 wiki 的 4–5 个基础文件填出来。

⌨️你输入什么
我要建一个 llm-wiki，让你以后每次回答前都先读一遍它。

现在先帮我"采访我自己"，把以下 4 个文件填出来：
- me.md（我的岗位、leader、汇报偏好、工作节奏）
- product.md（我的产品定位、关键模块、当前阶段、核心指标）
- glossary.md（公司/团队黑话词典，先 10 个）
- stakeholders.md（≥3 个核心协作方，他们在意什么）

外加一个面向行业周报场景的专属文件：
- competitor-snapshot.md（我长期盯的 3–5 家竞品快照：定位、最近动作、我家差异点）

采访方式：
1) 一次只问 1 个问题，等我回答再问下一个
2) 每个文件最多问 5 个问题，问到差不多就停
3) 全部问完后**先汇总给我看**，不要立刻写文件
4) 我确认后，再写到 pm-agent-lab/llm-wiki/ 下对应文件

约束：
- 不要替我编内容；我说"跳过"就跳过
- 文件内容简洁，每个 ≤ 200 字

2给 wiki 写一个"使用契约" README.md

没有契约的 wiki 等于没有 wiki——Agent 不会主动去读。
🎯你要做什么

4让 Agent 主动写回 wiki：建立"长记忆"

🌱 这是 wiki 真正强大的地方——它会自己长大

做一次"会议结论入 wiki"的演练。给 Agent 一组结论，让它先判断每条该不该入 wiki、入哪个文件、用什么写法，等你确认后再动文件。
⌨️你输入什么
我刚开完一个会，得到三条结论，请你判断哪条该入 wiki，哪条只是临时信息：

1) "下季度 OKR 重点放在留存提升，目标 D7 +2pp"
2) "周三下午改成跨部门同步会"
3) "leader 说他更喜欢看带具体数字的 bullet"

要求：
- 对每一条说明：进哪个文件 / 写哪一段 / 还是不入 wiki
- 给一份建议的写法（保留原意但更精炼）
- 不要立刻动文件，先给我看判断结果
我刚做完一轮需求池盘点，得到三条结论，请你判断哪条该入 wiki，哪条只是临时信息：

🧠 快速自测

Q1. 下面哪个最该进 Wiki？
正确答案 B：leader 偏好属于长期不变的"我是谁"事实——典型 me.md 内容。A 是临时变更（几周后失效）、C 是当下一次性意图（对话）、D 是重复流程（Skill）。
Q2. 「每周一跑一次新闻调研」应该是？
正确答案 B：重复性流程 → Skill。每周固定动作、固定步骤的工作，最适合沉淀成可复用的 Skill。Wiki 装的是"事实"，对话装的是"这一次的意图"。
Q3. 「下个月的产品迭代怎么排」？
正确答案 C：这是一次具体的"我现在要什么"——是当下意图，不是长期事实，也不是重复流程。讨论完之后，结论里的长期偏好（比如"以后都按 RICE 排"）才会被写回 wiki。

6思考题 & 下一讲引子

思考题

  • 如果你给同事看 me.md，他会不会觉得"挺懂你的"？如果不会，缺什么？
  • wiki 写得再好，遇到视觉信息（截图、设计稿、报错图）也无能为力——下一讲就解决这个。

下一讲引子

L4.5 我们让 Agent 看图：竞品截图怎么拆、设计稿改稿对比怎么写、报错图怎么解读——把 PM 每天看到的图都变成 Agent 可处理的输入。

8副线练习（可选）

本讲主线已经可在三个场景切换。副线建议把 wiki 拓展到具体业务方向：

  • 增加 competitors.md：你重点关注的 3–5 家竞品的定位、最近动作、自家差异点
  • 增加 metrics.md：你产品的核心指标定义、计算口径、目标值（注意红线：不要写未公开数字）
  • 增加 decisions.md：重大决策记录（每条 ≤ 5 行：背景 / 选项 / 决定 / 当时反对意见）
💡 decisions.md 是 PM 在 Agent 时代最被低估的资产：它能让 Agent 在你下次决策时主动提醒"上次类似情况你是这么决定的"。

9给 PM 的小结：Wiki / Skill / 对话 三件套

记住这张表，你就掌握了 Agent 时代 PM 的"个人知识架构"。

形态装什么例子寿命

Skill重复流程周报生成、新闻调研、需求池整理季度级
