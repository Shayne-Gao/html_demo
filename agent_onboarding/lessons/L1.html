让 Agent 第一次"动手"：操作本地文件

把 L0 那次"随手存"升级成结构化、可复用的本地数据，为 L2 做可视化打地基。

1你将学会

勾选完成情况会自动同步到首页学习进度。

  • 我能让 Agent 按我指定的目录、文件名、字段结构写入数据
  • 我能区分"自由文本"和"结构化数据"，并知道为什么 PM 应该优先要后者
  • 我能让 Agent 在写入前先预览方案，避免一上来就写错

2背景小知识

L0 你看到 Agent 能动手，但写出来的 markdown 表格——好看，但不好被下游消费。

PM 工作里有两种数据形态：

形态长什么样适合
自由文本一段话 / 一篇文档给人读
结构化数据JSON / CSV / 表格给程序读、给下一个 Agent 步骤读
💡 一个直觉口诀：先想清楚下游谁来用，再决定存什么格式。一旦你想"做可视化、做筛选、做月度对比、做自动化"，结构化数据就是必选项。

L2 我们要让 Agent 把这堆数据渲染成 HTML dashboard，所以本讲就要先把数据存对。

4实战任务

本讲我们在 pm-agent-lab/backlog-lab/ 下建立一个正经的数据目录，并教 Agent 按规范写入。

1先和 Agent 一起定 schema（字段定义）

💡 重要：这一步比直接喊"开始抓"重要 10 倍。没 schema 的 Agent，下游永远在返工。
🎯你要做什么

让 Agent 先给你一个字段定义草案，你来 review 和确认。

⌨️你输入什么
我要做一个"需求池整理"主线任务，每周把散落在飞书/邮件/访谈里的需求线索整理一次，存到本地，下游会用这些数据做 HTML 可视化。

帮我设计 backlog.json 的 schema（字段定义），要求：
1) 字段尽量原子化（一个字段只装一种信息）
2) 必含：需求描述、来源（具体到哪条 IM/邮件/访谈）、提出人、提出日期、所属模块、价值判断（1-5）、下一步动作
3) 给我两份产物：
   - schema/backlog.schema.md：字段名、类型、是否必填、示例值
   - 一条 mock 数据 sample.json，按 schema 长什么样

先不要去整理真实线索，先把这两份产物给我看。
我要做一个"会议纪要跟进"主线任务，每周把会议纪要里的待办与决策提取一次，存到本地，下游会用这些数据做 HTML 可视化。

帮我设计 follow-up.json 的 schema（字段定义），要求：
1) 字段尽量原子化（一个字段只装一种信息）
2) 必含：事项描述、归属人、紧急度（高/中/低）、截止日期、来源会议、会议日期、当前状态（待开始/进行中/已完成）
3) 给我两份产物：
   - schema/follow-up.schema.md：字段名、类型、是否必填、示例值
   - 一条 mock 数据 sample.json，按 schema 长什么样

我要做一个"行业新闻周报"主线任务，每周抓一次新闻，存到本地，下游会用这些数据做 HTML 可视化。

帮我设计 news.json 的 schema（字段定义），要求：
1) 字段尽量原子化（一个字段只装一种信息）
2) 必含：标题、来源媒体、原文链接、发布时间、行业标签、一句话摘要、重要性分（1-5）
3) 给我两份产物：
   - schema/news.schema.md：字段名、类型、是否必填、示例值
   - 一条 mock 数据 sample.json，按 schema 长什么样

先不要去抓真实新闻，先把这两份产物给我看。
👀预期看到什么
  • Agent 在 pm-agent-lab/backlog-lab/schema/backlog.schema.md 写一份字段表
  • 同目录或临时位置生成一条 sample.json 示例
  • 自然地停在这里等你确认
📝你要在这里做的事

认真 review schema：

  • 有没有字段是"两件事塞一起"的？比如"来源(标题)" → 拆成 source_name 和 source_title
  • 枚举字段是单选还是多选？（PM 视角：多选场景多，建议用数组）
  • 评分由谁打？是 Agent 自己评估的，还是你后续手工调整的？要不要再加一个 editor_note 字段？
这个对话本身就是 PM 工作的高频姿势：用自然语言定义数据契约。
⚠️如果出错怎么办
  • 偏差 A：Agent 直接开始抓数据了 → 纠偏：
    停一下。我现在不需要数据，只需要 schema 设计。请只产出 schema 和一条 mock sample.json，等我确认后再开始。
  • 偏差 B：schema 字段堆了 20 多个 → 纠偏：
    字段太多了。请按"一周后我用这些数据做 HTML dashboard"的最小必要集合精简到 8 个字段以内，其它放进 optional。

2让 Agent 抓本周数据并按 schema 写入

⌨️你输入什么
现在按已确认的 schema 整理本周（过去 7 天）散落在飞书/邮件/访谈里的需求线索 8–12 条，要求：
1) 严格按 schema 字段写入，缺失字段写 null，不要编造
2) 价值判断分（1-5）由你给，并把判断理由写进 editor_note
3) 写入路径：pm-agent-lab/backlog-lab/data/2026-W25/backlog.json
4) 同时生成一份 backlog.md（同样的内容，markdown 表格，给人读）
5) 写完告诉我：本次整理出几条、有几条因来源不清被丢弃、写入了哪些文件
现在按已确认的 schema 提取本周（过去 7 天）会议纪要里的待办与决策事项 8–12 条，要求：
1) 严格按 schema 字段写入，缺失字段写 null，不要编造
2) 紧急度由你给，并把判断理由写进 editor_note
3) 写入路径：pm-agent-lab/meeting-lab/data/2026-W25/follow-up.json
4) 同时生成一份 follow-up.md（同样的内容，markdown 表格，给人读）
5) 写完告诉我：本次提取出几条、有几条因归属人不明确被丢弃、写入了哪些文件
现在按已确认的 schema 抓本周（过去 7 天）AI Agent 行业新闻 8–12 条，要求：
1) 严格按 schema 字段写入，缺失字段写 null，不要编造
2) 重要性分（1-5）由你给，并把判断理由写进 editor_note
3) 写入路径：pm-agent-lab/news-lab/data/2026-W25/news.json
4) 同时生成一份 news.md（同样的内容，markdown 表格，给人读）
5) 写完告诉我：本次抓到几条、有几条因找不到链接被丢弃、写入了哪些文件
📅 本周 ISO 周编号：把 2026-W25 替换成实际值；不会算就直接告诉 Agent："按本周的 ISO 周编号命名目录"。
👀预期看到什么
  • 真的产生 .json 和 .md 两个文件
  • 打开 .json，每条记录字段齐全、JSON 格式合法
  • Agent 给你一个汇报式总结："抓到 11 条，丢弃 2 条（无来源链接），写入路径..."
⚠️如果出错怎么办
  • JSON 不合法：请用标准 JSON 格式重写，不要 // 注释、不要末尾逗号；写完用 python -m json.tool 验证一遍并告诉我结果。
  • 瞎编内容（最重要）：找不到真实来源/链接的条目直接丢弃，并在汇报里告诉我丢弃了几条、为什么。不要编造，这是硬要求。
  • 覆盖了上一周的目录：请确认你写入的路径是 data/<本周 ISO 周编号>/，不要覆盖其它周的数据。如果本周目录已存在，告诉我里面有什么，让我决定是覆盖还是合并。

3给数据加一层"质量自检"

💡 PM 不能等下游用数据时才发现质量问题。让 Agent 写完后自己 QA 一遍。
⌨️你输入什么
对刚写入的 backlog.json 做一次自检，按下面清单逐项给我答案：
1) 总条数 / 必填字段缺失的条数 / 价值判断分布
2) 有没有重复需求（描述高度相似或来源指向同一条线索）
3) 提出日期字段是否都在过去 7 天范围内
4) 所属模块里有没有明显错的、或者粒度不一致的（比如有的写"登录"有的写"账号体系"）
5) 给出 3 条改进建议，但不要直接动数据，先告诉我。
对刚写入的 follow-up.json 做一次自检，按下面清单逐项给我答案：
1) 总条数 / 必填字段缺失的条数 / 紧急度分布
2) 有没有重复事项（描述高度相似或来源指向同一次会议）
3) 截止日期字段是否合理（不应早于会议日期、不应过于久远）
4) 归属人字段里有没有缺失的、或者写法不一致的
5) 给出 3 条改进建议，但不要直接动数据，先告诉我。
对刚写入的 news.json 做一次自检，按下面清单逐项给我答案：
1) 总条数 / 必填字段缺失的条数 / 重要性分布
2) 有没有重复新闻（标题或链接重复）
3) 时间字段是否都在过去 7 天范围内
4) 行业标签里有没有明显错的、或者粒度不一致的
5) 给出 3 条改进建议，但不要直接动数据，先告诉我。
📝PM 决策时刻

看完报告，挑 1 个最值得修的问题，让 Agent 修：

把所属模块统一成下面这套受控词表：
[账号体系, 核心交易, 内容/社区, 增长/活动, 数据/后台, 平台/合规, 其他]
其它字段不要动。修完再跑一遍自检。
把归属人字段统一成下面这套受控写法：
[使用花名+所在团队，如"小明@增长"；待指派的统一写"待指派"]
其它字段不要动。修完再跑一遍自检。
把行业标签统一成下面这套受控词表：
[Agent 框架, Agent 应用, 模型能力, 投融资, 政策合规, 工具/CLI, 其他]
其它字段不要动。修完再跑一遍自检。
这就是「人 + Agent」协作的标准姿势：Agent 做体力活，PM 做判断和定义。

4把整套流程沉淀成一段"使用说明"

⌨️你输入什么（三场景通用）
把刚才整套流程写成一份"操作笔记"，存到当前 lab 目录的 data/2026-W25/HOWTO.md：
- 先列输入（覆盖的来源渠道、时间范围、目标条数）
- 再列步骤（schema → 抓取 / 整理 → 自检 → 修订）
- 每步贴出关键 prompt
- 末尾写一段"下周复用"指引：哪些要改、哪些可不改
💡 引子：这份 HOWTO.md 就是 L5 你要做 Skill 化的"前体"。先沉淀文字，再升级成 Skill，是非常稳健的路径。

5自检清单

  • 我打开 backlog.json，看到字段整齐、JSON 合法、内容真实
  • 我能向 Agent 一句话说清"为什么我要 schema 而不是直接给 markdown"
  • 我能在新会话里，仅靠 schema/*.schema.md + HOWTO.md，让 Agent 复现这一周的整理
  • 我已经主动做过 ≥1 次"让 Agent 先停下、先给方案、再执行"的 prompt 介入

🧠 快速自测

Q1. PM 为什么要 schema 而不是让 Agent 直接给 markdown？
正确答案 C：schema 的核心价值是为下游消费铺好"跑道"，让数据可被程序、可被下一个 Agent 步骤、可被未来的可视化工具直接用。
Q2. 你让 Agent 设计 schema 时，最该提醒它的是哪一条？
正确答案 B：让 Agent 先给方案再动手是 PM 必备姿势——一旦 schema 错了，下游一周白干。

6思考题 & 下一讲引子

思考题

  • 你的 schema 里有没有"该让 Agent 评分、但 PM 必须有覆写权"的字段？这种字段需要哪两列？
  • 如果你不是抓行业新闻/需求/纪要，而是抓"产品的用户反馈"，schema 至少要补哪几个字段？

下一讲引子

已经有了规整的本地数据，但 JSON 不适合人看。L2 会教你用一句话让 Agent 写一个可交互 HTML，把这些数据变成你和老板都爱看的 dashboard——而你不需要写一行代码。
